Design System · 2023

Verdure Design
System

A scalable component library and token system powering three products across two brands

The Challenge

Three product teams were making inconsistent UI decisions, leading to a fragmented experience and duplicated engineering effort. A shared system was needed that could flex across two distinct brand identities.

My Role

Design Systems Lead — token architecture, component design, documentation, Figma library, adoption strategy

Outcome

60% reduction in design-to-dev handoff time, 200+ components shipped, adopted by 4 teams

Tokens
first

The decision to start with tokens rather than components was deliberate and consequential. Without a well-structured token layer, any components we built would be brittle — tied to specific brand values and unable to flex across the two brand identities the system needed to serve.

We designed a three-tier token architecture: primitive tokens at the base (raw values like hex colours and pixel sizes), semantic tokens in the middle (purpose-driven names like `color.feedback.error`), and component tokens at the surface (scoped to specific patterns). This hierarchy gave us multi-brand theming almost for free — swapping a brand meant swapping a semantic token set, not rebuilding components.

Light and dark mode were built into the token architecture from the first day. Every colour token had a light and dark counterpart, defined in Figma variables and mirrored in the CSS custom property system delivered to engineers. There was never a moment when dark mode felt bolted on.

Components
that scale

With the token foundation in place, we built components using a strict atomic design methodology. Atoms — the smallest indivisible UI elements — were designed first, with every possible state documented: default, hover, active, focus, disabled, error. Only when an atom was fully specified did it graduate into molecules and organisms.

Accessibility was not a post-hoc consideration. Every component was designed with WCAG AA as a minimum, and tested with VoiceOver and NVDA before being marked complete. Focus states, colour contrast ratios, and keyboard interaction patterns were specified in the component documentation alongside visual specs.

Storybook served as the living documentation layer — each component story demonstrated every variant and state, with embedded accessibility annotations and usage guidelines. This gave engineers a single source of truth that stayed in sync with the Figma library through a token sync pipeline we built alongside the system.
